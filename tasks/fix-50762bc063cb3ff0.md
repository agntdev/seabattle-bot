# fix-50762bc063cb3ff0 — TOCTOU race in consumeInvite — two users can accept the same invite code

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 SBOT

`consumeInvite()` in `src/storage/invite-store.ts:76-88` reads the invite data and then deletes it in two separate, non-atomic operations. When two users open the same deep link simultaneously, both `consumeInvite` calls can read the invite data before either deletes it. Both will return `InviteData` (truthy), and the caller in `src/handlers/E5T1.ts:55-58` will proceed for **both** users, creating two pending matches from a single-use invite code. The second `createPendingMatch` will overwrite the first match entry (same `matchId`), silently dropping the first player's match while still notifying the second player.

Redis fix: use `GET`+`DEL` in a Lua script, or `SET` with `NX` and check return.

Fallback Map fix: use a synchronous check-and-delete pattern (JS is single-threaded per async context, but two `await`ed handlers can interleave).
